SyncFlow : Flux synchronisé pour une gestion logistique optimale

Overview

Mon projet d'outil de prise de commande tire son inspiration des besoins pratiques des clients, y compris les miens, ainsi que des exigences du personnel chargé de passer les commandes.

En prenant en compte ces diverses perspectives, j'ai cherché à développer une solution complète et fonctionnelle. . Les conseils et les retours des personnes qui m'ont encadré lors de mes immersions ont été particulièrement précieux dans ce processus de conception, m'aidant à identifier les fonctionnalités essentielles et les meilleures pratiques à intégrer dans mon produit.

Cette brève description met en évidence la démarche axée sur les besoins clients et l'importance des retours reçus lors de mes immersions dans le processus de développement de l'outil de prise de commande.

Architecture

Pour ce projet, j'ai délibérément choisi d'utiliser Java avec Spring Boot et MySQL. Lors d'une immersion professionnelle, on m'a expliqué que Java était une technologie de choix en raison de sa fiabilité, de sa portabilité et de sa robustesse, en plus de ses capacités à gérer des logiques complexes propres au backend et à la gestion des données. Cette expérience m'a convaincu que Java serait une excellente option pour ce projet. De plus, j'ai été influencé par la grande communauté de développeurs autour de Java, ce qui m'a donné confiance dans ma décision. J'ai également utilisé MySQL Workbench pour concevoir ma base de données de A à Z, me permettant de comprendre pleinement l'architecture d'une base de données et de mettre en pratique mes connaissances acquises lors de mes immersions professionnelles.

Pour le frontend, j'ai opté pour Vue.js en raison de sa simplicité d'utilisation, ce qui m'a permis de créer rapidement une interface utilisateur conviviale pour mon projet. En outre, j'ai pu bénéficier de l'expertise de la communauté Java lorsque j'avais besoin de retours spécifiques concernant Java, ce qui a grandement contribué au succès de mon projet. J'ai ainsi pu tirer parti des ressources disponibles pour résoudre des problèmes et améliorer continuellement mon code, démontrant ainsi ma capacité à utiliser efficacement les ressources communautaires pour soutenir mon travail de développement logiciel. De plus, j'ai utilisé Postman et Soap UI.

Stratégie d'organisation et méthodologie de développement

Gestion de la collecte et du traitement des données en temps réel

Ma stratégie de développement a débuté par l'établissement d'une connexion solide entre le backend et la base de données, jetant ainsi les bases de mon projet. Cette fondation m'a permis de récupérer et d'afficher les données sur l'interface frontend, conformément au design défini dans le wireframe initial. Pour garantir la qualité et la fiabilité de chaque étape du processus, j'ai adopté une approche itérative, testant régulièrement le backend via le terminal et des requêtes Curl, tout en assurant un suivi efficace grâce à un système de journalisation intégré dans mon projet Spring Boot.

Face aux défis rencontrés, je suis resté focalisé sur mon plan d'action initial, demeurant concentré sur l'idée de base du produit, notamment sa version initiale (V1) dans Figma.

Les parties jaunes et vertes représentaient les priorités que je m'étais fixées, et c'est précisément ce que j'ai développé dans ma V1. J'ai adopté une approche méthodique, préservant le code précédent par des commentaires tout en développant une nouvelle logique conforme aux besoins métier émergents. En parallèle du développement, j'ai surveillé attentivement les performances de l'interface web, en inspectant les outils de développement du navigateur pour détecter les problèmes potentiels.

Les logs (IMAGE CI-DESSOUS) ont joué un rôle crucial en me fournissant une visibilité claire sur les erreurs rencontrées, ce qui m'a permis de les résoudre de manière efficace.

Aussi, j'ai accédé à mon serveur local pour vérifier le bon fonctionnement de l'API que j'avais développée, afin de m'assurer qu'elle récupérait correctement les données de ma base de données (RESULTAT CI-DESSOUS). . À chaque fois que j'ai réussi à résoudre une erreur, j'ai consigné la solution dans un fichier dédié que j'ai conservé dans mon projet. De plus, j'ai maintenu une gestion rigoureuse des versions en enregistrant les différentes itérations du projet dans des dossiers zippés sur mon ordinateur, assurant ainsi une traçabilité claire de son évolution.

J'ai également mis en place le Domain-Driven Development (DDD) dans mon projet pour organiser les packages par domaine.

Cette façon de faire m'a été conseillée par mon mentor de projet, Thomas Wilgenbus, un expert en Java que j'ai choisi pour son expertise dans ce domaine. Cette approche méthodique et itérative a permis une progression efficace dans le développement de mon outil de prise de commande, en garantissant sa qualité, sa fiabilité et sa conformité aux besoins spécifiques des utilisateurs.

Validation et tests de qualité du code

Tests unitaires des fonctionnalités principales

Tests de contrôleurs REST :
Ces tests vérifient que mes contrôleurs, qui gèrent les requêtes HTTP, renvoient les bonnes réponses. J'ai effectué ces tests en utilisant des outils tels que Postman, SOAP UI, ainsi que directement depuis mon terminal et mon backend. Par exemple, je vérifie que lorsque j'envoie une requête pour récupérer un utilisateur, le contrôleur renvoie bien les informations de cet utilisateur.

Tests de validation :
Ces tests vérifient que les règles de validation que j'ai définies pour mes modèles sont appliquées correctement. Je vérifie que les contraintes de validation sont respectées et que les erreurs sont correctement gérées. Par exemple, je vérifie que lorsque j'essaie de créer un nouvel utilisateur avec un email invalide, l'application renvoie un message d'erreur approprié.

Tests de persistance :
Ces tests vérifient que mes classes Repository, qui interagissent avec la base de données, fonctionnent correctement. J'ai effectué ces tests en utilisant des commandes curl directement depuis mon terminal pour simuler les appels à la base de données et vérifier les résultats. Par exemple, je vérifie que lorsque j'appelle la méthode pour récupérer un utilisateur dans le repository, celui-ci renvoie bien les bonnes données depuis la base de données.

Tests de l'API développé :
Pour tester ma propre API, j'ai utilisé des outils tels que Postman, cURL et des requêtes HTTP directement depuis mon terminal. J'ai commencé par définir les cas de test pour chaque fonctionnalité de mon API, comme la création, la lecture, la mise à jour et la suppression des ressources. En envoyant des requêtes HTTP à mon API, j'ai vérifié les réponses renvoyées pour m'assurer qu'elles étaient conformes à mes attentes. J'ai également testé le comportement de mon API dans des scénarios d'erreur en envoyant des requêtes avec des données incorrectes et en vérifiant les réponses d'erreur correspondantes. Pour faciliter le processus de test, j'ai envisagé d'automatiser certains tests en utilisant des outils de test automatisés comme JUnit ou RestAssured. Cela me permettrait de répéter rapidement mes tests à chaque fois que je fais des modifications à mon API.
J'ai également effectué des requêtes cURL telles que : curl http://localhost:8080/parametres/all pour vérifier si mon API recevait correctement les données de chaque table de ma base de données.

Tests d'intégration :
Ces tests vérifient que tous les composants de mon application fonctionnent correctement ensemble. Je teste les interactions entre les contrôleurs, les services et les repositories pour m'assurer qu'ils communiquent correctement entre eux. Par exemple, je vérifie que lorsque j'envoie une requête HTTP, celle-ci déclenche correctement les opérations nécessaires dans les services et les repositories pour traiter cette requête.

En parallèle, j'ai utilisé Postman et SOAP UI pour simuler les appels à l'API REST et aux services web SOAP, respectivement. Cela m'a permis de valider le bon fonctionnement de ces services et d'identifier les éventuels problèmes d'intégration ou de communication. J'ai également maintenu un fichier de suivi sur Google Meet dans lequel j'ai consigné toutes les erreurs rencontrées et les solutions que j'ai apportées pour les résoudre, incluant les aspects liés à la base de données, au backend et au front-end. Cette pratique m'a aidé à garder une trace de mes découvertes et à capitaliser sur mes expériences pour éviter de reproduire les mêmes erreurs à l'avenir.

Enfin, j'ai consigné l'ensemble de mes commandes dans un fichier shell, ainsi que dans mon document Google Meet.
